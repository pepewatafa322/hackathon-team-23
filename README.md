# hackathon-team-23
Хакатон. Автоматизированная система обработки корпоративной почты. Команда 23

Проект разработан для автоматизации работы IT-поддержки. Система сканирует папку входящих писем inbox/, анализирует их контент, классифицирует по 10 категориям и автоматически раскладывает по соответствующим папкам, генерируя итоговый текстовый отчет и инфографику в виде HTML-дашборд.

## Схема работы пайплайна:
1. EmailReader сканирует папку inbox/, отсекает бинарные файлы и картинки, подбирает кодировку и  далее читает текст.
2. EmailParser распознает один из 4 форматов заголовков или парсит JSON-структуру, создавая единый объект класса Email.
3. RuleBasedClassifier проверяет письмо по набору правил RULES, отсортированных по приоритету.
4. EmailCopier создает структуру папоr и копирует файлы в папки соответствующих категорий.
5. ReportGenerator сохраняет общую текстовую статистику в report.txt c процентной долей встречаемости, а также в report.html с интерактивным дашбордом.


## Категорий классификации
1. incident - критические сбои, аварии, падение баз данных (приоритет 80).
2. monitoring - автоматические уведомления систем (приоритет 90).
3. spam - нежелательные письма, реклама, фишинг и тд (приоритет 100).
4. hardware - проблемы с клавиатурами, мышами, принтерами и тд (приоритет 60).
5. access - запросы прав, VPN, учетных записей (приоритет 50).
6. documents - акты, счета, договоры, закрывающие документы (приоритет 45).
7. hr - отпуска, больничные, графики работы (приоритет 40).
8. request - установка ПО, мелкие баги, настройка систем (приоритет 30).
9. internal - совещания, созвоны, дайджесты (приоритет 20).
10. unclassified - пустые, нечитаемые или не подошедшие под правила письма (приоритет 0).

## Устойчивость классификации
Безопаcность:
   Если письмо не подходит ни под одно правило (или файл пустой/бинарный), оно никогда не теряется. Система сохраняет его в папку unclassified/, делает предупреждающую запись в лог с указанием причины.
Гибкая ООП-архитектура правил:
   Все правила вынесены в класс ClassificationRule в файле rules.py. Если в компании появится новая категория писем, разработчику не нужно менять логику программы, парсера или классификатора. Достаточно просто добавить один объект правила в общий список RULES в rules.py.


## Расширенное решение
В качестве рассширенного решения мы предлагаем dashboard со статистекой для бизнеса-аналитики:
1. Временная диаграмма, показывающая распределение сообщений во время дня, для планирования смен поддержки.
2. Топ самых активных отправителей, который помогает выявить самых «проблемных» авторов обращений или спам-ботов в реальном времени.
3. Статистика распределения писем по категориям в виде диограммы.

Инструкция для запуска:
1. pip install -r requirements.txt (там только pytest)
2. запустить в терминале в папке с проектом ./run.sh (запуск bush скрипта)
или python -m src.main --inbox inbox --output processed (простой запуск через питон)
